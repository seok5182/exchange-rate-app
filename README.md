# 💱 Exchange Rate App (환율 정보 조회 앱)

시니어 개발자 스타일의 관심사 분리(SoC)를 적용한 환율 정보 크롤러/API 클라이언트 프로젝트입니다. 단순히 동작하는 코드를 넘어, 유지보수와 보안을 고려한 아키텍처를 지향합니다.

## 📂 Project Structure
- `src/api/`: 외부 API 통신 전담 (Data Layer). 오직 데이터를 가져오는 일에만 집중하며, 통신 에러를 1차적으로 관리합니다.
- `src/services/`: 비즈니스 로직 전담 (Service Layer). 가져온 Raw 데이터를 앱의 목적에 맞게 가공(요리)합니다.
- `src/config/`: 환경 변수 및 설정값 관리 (Config Layer). `appConfig.js`를 통해 시스템 전체 설정을 중앙 관리합니다.
- `src/utils/`: 공용 유틸리티 (Utility Layer). 날짜 포맷팅 등 범용적인 기능을 독립적으로 관리합니다.
- `src/app.js`: 애플리케이션 진입점 (Entry Point). 사용자 입력을 받고, 각 레이어의 부품을 조립하여 실행을 지휘합니다.

## 💡 Knowledge Base (상세 학습 로그)

### 1. 데이터의 기본 단위: 변수와 객체
- **변수 (Variable)**: 단일 데이터를 저장하는 최소 단위. 이름표가 붙은 개별 포스트잇과 같습니다.
- **객체 (Object)**: 연관된 여러 변수를 하나로 묶은 데이터 꾸러미입니다. 이름표가 붙은 칸이 여러 개 있는 수납함과 같습니다.
- **자바스크립트 객체 = Map**: 자바스크립트의 일반 객체(`{}`)는 Java의 **`Map` (특히 `HashMap`)**과 사실상 동일한 Key-Value 구조를 가집니다.
  - **문법 차이**: Java는 `get()`, `put()` 메서드를 호출하지만, JS는 **`[]` (대괄호)** 문법을 통해 직접 접근합니다.
- **저장 방식의 차이 (핵심!)**:
  - **직접 저장 (Value)**: 문자열이나 숫자 같은 단일 정보는 변수에 값이 직접 담깁니다.
  - **주소 저장 (Reference)**: 객체는 메모리에 생성되며, 변수는 그 객체의 **'주소(리모컨)'**를 담고 있습니다. 리모컨을 복사해도 결국 같은 TV를 조종하게 됩니다.
- **객체 중첩 (Nested Object)**: 객체 내부 속성으로 또 다른 객체를 담아 데이터를 계층화합니다. (예: `result.rates.KRW`)

### 2. ES Modules 및 Import 규칙
- **`type: module`**: Node.js에서 최신 표준 문법(`import/export`)을 사용하기 위한 필수 설정입니다.
- **확장자 필수**: ES Modules 환경에서는 `import` 시 반드시 **`.js` 확장자**를 명시해야 합니다. 이를 누락하면 `ERR_MODULE_NOT_FOUND` 에러가 발생합니다.
- **따옴표 일관성 (Quotes)**: 작은따옴표(`'`)와 큰따옴표(`"`)는 기능적으로 동일하지만, 프로젝트 전체에서 하나로 통일하여 사용하는 것이 가독성과 유지보수 측면에서 중요합니다. (본 프로젝트는 작은따옴표를 지향합니다.)

### 3. 호이스팅(Hoisting)과 함수 설계
- **호이스팅**: `function` 선언문이 코드 최상단으로 끌어올려지는 현상. 코드 흐름을 예측하기 어렵게 만듭니다.
- **안전한 설계**: `const` 화살표 함수는 호이스팅을 방지하여 코드가 반드시 선언된 이후에만 사용되도록 강제합니다. 이는 예측 가능한 안전한 코드를 만드는 시니어의 기술입니다.

### 4. 매개변수 설계 및 데이터 정제 (Normalization)
- **Named Parameters**: 인자를 객체`{}`로 묶어 전달하면, 순서와 상관없이 의도가 명확해지고 필요한 값만 선택적으로 넘길 수 있는 고급 설계 기법입니다.
- **Default Trigger**: 기본값은 해당 속성이 `undefined`일 때 발동합니다.
- **Double Defaulting (이중 안전장치)**: `{ key = value } = {}` 문법은 에러 방지와 유연성을 동시에 확보합니다.
  - **작동 원리**: 인자 자체가 누락되면 뒤쪽의 `= {}`가 빈 객체를 먼저 생성합니다. 이후 이 빈 객체가 앞쪽의 `{ key = value }`와 결합하면서 모든 속성이 기본값으로 채워진 완성된 객체가 됩니다.
  - **이점**: 인자를 아예 넣지 않고 호출해도 에러 없이 기본 로직이 실행되도록 보장합니다.
- **Configuration Object (설계 철학)**: `Intl` API나 우리가 만든 `getSummary`처럼, 설정값이 많을 때는 이를 하나의 **객체`{}`**로 묶어 전달합니다.
- **Required vs Optional (하이브리드 설계)**: `Intl` API처럼 필수 정보(예: locale)는 독립된 인자로 빼고, 나머지 부가 옵션만 객체로 받는 방식도 널리 쓰입니다. 이는 **"필수값"**과 **"선택값"**을 시각적으로 명확히 구분해주는 효과가 있습니다.
- **일관성 있는 정제**: 시스템 입구(`index.js`)에서 모든 문자를 대문자로 변환하여 데이터 일관성을 확보합니다.
- **Interactive CLI**: `readline` 모듈을 통해 실행 중 사용자 질문을 던져 프로그램의 유연성을 극대화합니다.

### 5. 객체/배열 가공 테크닉 및 디버깅
- **객체 접근법**: 정적 접근은 `Dot(.)`, 변수를 이용한 동적 접근은 **`Bracket([])`**을 사용합니다. 환율 데이터처럼 키 이름을 변수로 받아올 때는 대괄호 문법이 필수입니다.
- **구조 분해 할당 (Destructuring)**: `const { rates } = result;` 처럼 필요한 정보만 우아하게 추출합니다.
- **구조 분해의 함정 (디버깅)**: `const { rates } = data.rates`와 같이 이미 파고든 객체에서 동일한 이름의 속성을 찾으려 하면 `undefined` 에러가 발생합니다. 항상 데이터의 계층 구조를 먼저 확인해야 합니다.
- **런타임 에러 읽기**: `Cannot read properties of undefined` 메시지는 부모 객체가 비어있음을 뜻합니다. 이를 통해 어느 지점에서 참조가 끊겼는지 추적할 수 있습니다.
- **객체 반복 처리 (Object.entries)**: 객체를 배열(`[[K,V]]`)로 변환하여 `for...of` 문에서 처리할 수 있게 해줍니다.
- **배열 가공 테크닉 (Method Chaining)**: `split(',').map(...)` 처럼 메서드를 연결하여 데이터를 단계별로 정제(Sanitization)합니다.
- **Utility Layer (유틸리티)**: 날짜 가공과 같이 시스템 전반에서 공통으로 사용되는 로직을 `src/utils`에 분리하여 코드의 **재사용성**을 극대화합니다.
- **국제화 API (Intl)**: `Intl.DateTimeFormat`과 같은 내장 도구를 사용하면 언어별 날짜 표준 형식을 코드를 직접 조립하지 않고도 우아하게 구현할 수 있습니다. (확장성 확보)

### 6. 데이터 통신의 원리 (JSON & Axios)
- **직렬화/역직렬화**: 인터넷은 문자열만 나르며, `axios`는 이를 객체로 자동 복원해줍니다.
- **API Exploration**: 코드를 짜기 전 반드시 `console.log`로 실제 데이터의 모양을 확인해야 정확한 설계가 가능합니다.
- **JSDoc**: 타입 명시가 없는 JS에서 주석으로 타입을 설명합니다. 단, 실제 데이터와 대조하여 검증하는 과정이 필수적입니다.

### 7. 네트워크 보안 (CORS & CSRF)
- **CORS**: 서버가 제시한 허용 명패를 브라우저가 확인하여 데이터를 차단하는 보안 정책입니다. 서버 간 통신은 제약이 없습니다.
- **CSRF**: 브라우저의 쿠키 자동 전송 특성을 악용한 공격입니다. 서버 사이드 통신은 상대적으로 안전합니다.
- **인터셉터 (Interceptors)**: 
  - 요청/응답의 관문으로, 결과를 뒤바꿀 수 있는 강력한 권한을 가집니다.
  - **계층 구조의 한계**: 브라우저 보안 엔진이 CORS 위반으로 데이터를 파기하면, 인터셉터조차 그 데이터를 가로챌 수 없습니다.

### 8. 환경 변수 및 Config 레이어
- **보안과 설정 분리 (.env)**: API URL이나 키와 같은 민감한 정보는 `.env` 파일에 저장하여 Git 관리에서 제외합니다. (금고 역할)
- **Config 레이어의 역할 (src/config)**: `.env`의 값을 읽어와 자바스크립트 객체로 가공하여 관리합니다. 코드는 금고(`.env`)를 직접 뒤지는 대신, 잘 정리된 지갑(`config` 객체)에서 값을 꺼내 쓰게 됩니다. 이는 타입 안정성과 중앙 집중 관리를 위한 시니어의 설계 방식입니다.
- **네이밍 전략 (Naming Strategy)**: 
  - `index.js`라는 이름이 중복되어 혼란을 줄 경우, 진입점 파일을 `app.js`나 `main.js`로 명명하여 역할(EntryPoint)을 강조합니다.
  - `baseUrl`처럼 범용적인 이름 대신 `exchangeApiUrl`과 같이 구체적인 이름을 사용하여, 향후 다양한 서비스가 추가되더라도 설정값이 충돌하지 않도록 확장성을 고려합니다.
