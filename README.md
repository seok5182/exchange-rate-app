# 💱 Exchange Rate App (환율 정보 조회 앱)

시니어 개발자 스타일의 관심사 분리(SoC)를 적용한 환율 정보 크롤러/API 클라이언트 프로젝트입니다. 단순히 동작하는 코드를 넘어, 유지보수와 보안을 고려한 아키텍처를 지향합니다.

## 📂 Project Structure
- `src/api/`: 외부 API 통신 전담 (Data Layer). 오직 데이터를 가져오는 일에만 집중합니다.
- `src/services/`: 비즈니스 로직 전담 (Service Layer). 가져온 데이터를 우리 앱에 맞게 가공(요리)합니다.
- `src/index.js`: 애플리케이션 진입점 (Entry Point). 모든 부품을 조립하고 실행을 지휘합니다.

## 💡 Knowledge Base (상세 학습 로그)

### 1. ES Modules 및 Import 규칙
- **`type: module`**: Node.js에서 최신 표준 문법(`import/export`)을 쓰기 위한 설정입니다.
- **확장자 필수 규칙**: ES Modules 환경에서는 파일을 가져올 때 반드시 **`.js` 확장자**를 명시해야 합니다. (예: `import { func } from './file.js';`) 이를 누락하면 `ERR_MODULE_NOT_FOUND` 에러가 발생합니다.

### 2. 호이스팅(Hoisting)과 안전한 설계
- **`function` 선언식**: 선언하기도 전에 호출할 수 있는 호이스팅이 발생합니다. 이는 편리해 보이지만 코드가 복잡해질 때 예측 불가능한 에러를 만듭니다.
- **`const` 화살표 함수**: 반드시 선언한 이후에만 호출 가능합니다. 코드가 위에서 아래로 흐르는 **논리적 순서**를 강제하여 훨씬 안전한 코드를 만듭니다.

### 3. 매개변수 설계와 하드코딩 지양 (Avoid Hardcoding)
- **Default Parameter**: `async (baseCurrency = 'USD')` 처럼 기본값을 주면 사용자는 편하게 쓰면서도, 필요할 때 언제든 기준 통화를 바꿀 수 있는 **유연성**을 갖게 됩니다.
- **철학**: 서비스 로직 내부에 'USD'라는 글자를 직접 박아넣지 마세요. 나중에 요구사항(예: 유로 기준 조회)이 바뀌었을 때 코드 내부를 건드리지 않고 호출부만 수정하기 위함입니다.

### 4. 변수 선언과 객체/배열 다루기
- **`const` vs `let` (쇼핑 카트 비유)**:
  - `const`는 **'집 주소'**를 고정하는 것입니다. 집 안의 가구를 바꾸는 것(객체 내부 수정)은 허용되지만, 아예 다른 집으로 이사하는 것(변수 재할당)은 금지됩니다.
  - 실무에서는 일단 무조건 `const`로 시작하고, 재할당이 꼭 필요한 순간에만 `let`으로 바꿉니다.
- **객체 접근법 (Dot vs Bracket)**:
  - **Dot (`obj.key`)**: 이름을 정확히 알 때 씁니다.
  - **Bracket (`obj[variable]`)**: 변수에 담긴 값을 통해 동적으로 접근할 때 필수입니다. 환율 데이터처럼 어떤 통화 코드(`currency`)가 올지 모를 때는 반드시 대괄호 표기법을 써야 합니다.
- **구조 분해 할당 (Destructuring)**: `const { rates } = data;` 처럼 객체에서 필요한 것만 쏙 뽑아 쓰는 방식입니다. 여러 줄의 할당 코드를 한 줄로 줄여 가독성을 비약적으로 높여줍니다.

### 5. 데이터 통신의 원리 (JSON & Axios)
- **직렬화/역직렬화**: 인터넷은 오직 '문자열'만 실어 나릅니다. 서버가 보낸 긴 JSON 문자열을 `axios`가 자동으로 자바스크립트 객체로 조립(역직렬화)해 주기에 우리가 편하게 데이터를 다룰 수 있습니다.
- **API Exploration**: 시니어 개발자는 코드를 짜기 전 반드시 **`console.log(data)`**를 찍어봅니다. 데이터가 객체인지 배열인지 눈으로 확인해야만 정확한 로직을 설계할 수 있기 때문입니다.

### 6. 브라우저 보안의 핵심 (CORS)
- **제시와 집행**: **서버**는 허용 명단(`Allow-Origin`)을 제시하고, **브라우저**는 이를 확인하여 데이터를 파기할지 결정(집행)합니다.
- **Simple vs Preflight**:
  - 일반 요청은 일단 보내고 응답을 보며 판단합니다.
  - 수정/삭제 같은 위험한 요청은 `OPTIONS` 메서드로 미리 "해도 돼?"라고 물어보고(Preflight), 허락되지 않으면 본 요청을 보내지도 않습니다.
- **Node.js의 자유**: 서버-서버 통신은 브라우저 같은 보안 요원이 없으므로 CORS 제약 없이 자유롭게 통신합니다.

### 7. 보안 위협과 방어 (CSRF & Interceptors)
- **CSRF (사이트 간 요청 위조)**: 브라우저가 목적지 도메인에 맞는 쿠키(인증 정보)를 자동으로 실어 보내는 특성을 악용한 공격입니다. 로그인 탭을 껐더라도 세션이 살아있으면 나쁜 사이트가 내 권한을 이용해 요청을 보낼 수 있습니다.
- **인터셉터 (Interceptors)**:
  - 요청/응답의 **양방향 관문**입니다. 성공을 실패로, 실패를 성공으로 뒤바꿀 수 있는 강력한 권한을 가집니다.
  - **계층 구조의 한계**: 인터셉터는 자바스크립트 계층이므로, 더 상위 계층인 브라우저 네트워크 엔진이 CORS 위반으로 데이터를 파기해버리면 그 데이터를 가로채거나 조작할 기회조차 갖지 못합니다.
